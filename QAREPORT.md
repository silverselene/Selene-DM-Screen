## 📋 Summary

- Adds the optional local AI bridge (`services/ai-bridge`: Claude Agent SDK + ddb-mcp, SSE streaming), the AI Chat widget (history persistence, tool cards, model/effort picker, local lookup), the Portal widget, and a types-only `bridge-protocol` wire contract.
- Unifies the monster dataset (2,146 of 2,160 entries now carry full stat blocks), expands the compendium 8×, adds spell damage summaries, and wires two Claude GitHub workflows.
- Agents were consistently impressed by the engineering discipline: origin-allowlisted CORS, deny-by-default DDB tool allowlist with regression tests, injection-safe markdown renderer, atomic two-phase backup import, strong pure-logic test coverage.
- The blocking problem is the bridge's disconnect handling, which is **broken in both halves on Node 24** — verified twice by independent reproduction.

### Key findings

- 🔴 Bridge never notices client disconnects: `req.on("close")` never fires and `res.writable` never goes false — Stop/reload leaves Agent SDK turns running to completion, silently spending the Claude subscription; stop-and-retry stacks unbounded concurrent turns (reproduced empirically, twice).
- 🟠 Chat transcript is count-capped but byte-unbounded — 5 agents independently converged on the same failure chain: localStorage quota pressure on shared keys (incl. mid-encounter Initiative) + backups that export fine but silently skip the transcript on restore (2 MB import cap).
- 🟠 Portal YouTube passthrough hosts (`youtube.com`, `m.youtube.com`, bare `youtube-nocookie.com`) are not in the Docker CSP `frame-src` — links work in dev, render a silent blank iframe in the container (3 agents).
- 🟠 Hardcoded port/origin constants in three places make every documented alternate deploy (`PORT`, `AI_BRIDGE_PORT`, reverse proxy) fail as a misleading "bridge not running" (3 agents).
- 🟠 Spell tool cards silently substitute bundled 2024 data for the actual D&D Beyond result on any name match — wrong rules text with no provenance hint (3 agents).
- 🟡 The bridge's "no filesystem/exec" guarantee rests solely on `canUseTool`, which may not intercept auto-permitted read-only built-ins — prompt injection in DDB content could read local files (security).

---

## 🏁 Verdict

> ⚠️ **REQUEST CHANGES**

One agent returned HIGH (reliability, with an empirical repro the synthesizer independently confirmed) alongside six MEDIUMs → overall **HIGH**. The disconnect-handling fix is two lines (`res.on("close")` + `res.destroyed`); the transcript byte cap and the Portal/CSP host normalization should land with it. Everything else is small, local hardening.
